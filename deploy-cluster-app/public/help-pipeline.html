ZTD Deployment Pipeline

Full end-to-end automation for Nutanix cluster deployments — from bare-metal iLO boot through Phoenix OS imaging, cluster creation, post-deployment configuration, and NCC health validation. A single JSON config file drives the entire pipeline.

PowerShell 7 required Sequential multi-phase pipeline Resume from any step Dry-run mode Real-time WebSocket streaming

Overview

Start-Pipeline.ps1 orchestrates the full deployment lifecycle. It accepts a single JSON configuration file and runs up to 18 sequential steps. A step failure immediately halts the pipeline — use -StartAtStep N to resume from the failure point without re-running completed steps.

The pipeline is designed to be triggered from the ZTD Deployment Manager web app, but can also be run directly from a PowerShell 7 session on the jump server.

Key capabilities

  • Pre-flight validation — checks iLO access, PC credentials, IP availability, URL accessibility, and DNS reachability before any destructive action
  • Dry-run mode (-DryRun) — runs only pre-flight checks, no steps execute
  • What-if preview (-WhatIf) — prints each step with arguments, nothing executes
  • Resume & skip — -StartAtStep N skips earlier steps; -SkipSteps "7,9" skips individual steps
  • Optional steps — Steps 14 (LCM Inventory) and 18 (NCC Health Check) auto-skip unless enabled in config
  • Pipeline result email — HTML email with step timings, NCC health summary, and LCM report on completion

Architecture

The diagram below shows how components relate across the jump server, HUB site, remote site, and corporate infrastructure. The jump server orchestrates everything; Foundation Central and Prism Central do the heavy lifting for imaging and configuration.

Jump server orchestrates all 18 steps. Foundation Central and Prism Central run on the HUB site but reach the remote site over the WAN.

⚠ FC and PC must reach the remote site. From the Foundation Central VM and Prism Central VM (running on the HUB cluster), the iLO VLAN and AHV/CVM Management VLAN at the remote site must be routable and accessible before starting the pipeline. Engage the network team to verify inter-site routing before deployment.

Network infrastructure (configured by network team)

  • AHV/CVM Management VLAN must be Access/Native on server-facing switch ports
  • All production VLANs must be Tagged on the same switch ports
  • Switch ports must use LACP no-suspend — uplinks stay active before AHV bonding is configured
  • iLO/BMC ports must be connected and tagged on the iLO VLAN, reachable from the jump server before pipeline starts

DHCP and Foundation Central API key

Foundation Central auto-discovers nodes only when the FC API key is embedded in DHCP vendor-specific options (option 43, vendor class NutanixNode) on the AHV management VLAN. Without this, nodes boot Phoenix but never appear in Foundation Central.

📖
Foundation Central — DHCP Vendor-Specific Options for Nodes
portal.nutanix.com — Official guide for configuring option 43 so nodes are auto-discovered by FC

Custom Phoenix image — per region requirement

Phoenix ISO images must be generated from your HUB site's Foundation Central API key. A generic upstream Phoenix ISO will not carry the correct FC API key — nodes boot Phoenix but are never discovered by FC.

  1. In Foundation Central → Settings → API Keys — create or copy the region API key
  2. Generate the custom Phoenix ISO using the HUB cluster CVM (see official guide below)
  3. Upload the ISO to Azure Blob Storage and use that direct URL as phoenix_iso_url in the config
⚠ One Phoenix ISO per region. A Phoenix image generated for one region's HUB will not work for a different region's sites. Each region requires its own image.
📖
Foundation Central v1.10 — Generating the Custom ISO URL
portal.nutanix.com — Step-by-step guide for generating a site-specific Phoenix ISO from the HUB CVM

Prism Central (must already be running)

  • PC must be deployed and accessible at the IP in the config (prism_central.ip)
  • All production VLANs must already be configured as networks in PC
  • Foundation Central VM must be deployed and reachable from PC and from the jump server (port 9440)
  • Protection/recovery policies and categories referenced in the config must already exist in PC

DNS service account (Step 17)

Step 17 creates DNS A records for all node hostnames and the cluster VIP using dnscmd.exe with credential delegation. Requirements:

  • A domain service account (e.g. CORP\SVC-NTX-AUTO) with Create/Delete DNS records permission on the company.net and company.ilo zones
  • dnscmd.exe installed on the jump server (part of DNS RSAT tools)
  • Service account credentials set under dns_admin in the config
Step 17 is automatically skipped if dns_admin.username or dns_admin.password are absent from the config — no error is raised.

Quick Start

  1. 1

    Create a configuration file

    Copy the example template and edit it with your cluster's IPs, credentials, image URLs, and VLAN settings.

    Copy-Item .\Configs\example_template.json .\Configs\MY-CLUSTER-01.json
notepad .\Configs\MY-CLUSTER-01.json
  2. 2

    Validate with dry-run

    Runs all pre-flight checks — iLO access, PC credentials, IP availability, URL accessibility. No pipeline steps execute. Safe to run anytime.

    .\Start-Pipeline.ps1 -ConfigFile .\Configs\MY-CLUSTER-01.json -DryRun
  3. 3

    Preview the plan

    Prints each step with its script path and arguments. Nothing executes.

    .\Start-Pipeline.ps1 -ConfigFile .\Configs\MY-CLUSTER-01.json -WhatIf
  4. 4

    Run the full pipeline

    Starts pre-flight checks, then executes all 18 steps in sequence.

    .\Start-Pipeline.ps1 -ConfigFile .\Configs\MY-CLUSTER-01.json
  5. 5

    Resume after a failure

    Use -StartAtStep to skip already-completed steps. Combine with -SkipSteps to skip individual steps.

    # Resume from step 7 (steps 1–6 already completed)
.\Start-Pipeline.ps1 -ConfigFile .\Configs\MY-CLUSTER-01.json -StartAtStep 7

# Skip step 9 (already exists), resume from step 8
.\Start-Pipeline.ps1 -ConfigFile .\Configs\MY-CLUSTER-01.json -StartAtStep 8 -SkipSteps "9"

# Resume but skip pre-flight (IPs known-good)
.\Start-Pipeline.ps1 -ConfigFile .\Configs\MY-CLUSTER-01.json -StartAtStep 5 -SkipPreCheck

Common Examples

Run a dry-run with verbose output

.\Start-Pipeline.ps1 -ConfigFile .\Configs\MY-CLUSTER-01.json -DryRun

Run with a notification email

The notification recipient is normally read from notify.to in the config. You can override it using -TriggeredBy:

.\Start-Pipeline.ps1 -ConfigFile .\Configs\MY-CLUSTER-01.json `
    -TriggeredBy "engineer@company.com" -Cc "manager@company.com"

Run NCC health check only

Run Step 18 standalone against an existing cluster without running the full pipeline:

.\Run-SystemHealthCheck.ps1 -ConfigFile .\Configs\MY-CLUSTER-01.json

Run password hardening only

.\Change-Prism-CVM-AHV-Password.ps1 -ConfigFile .\Configs\MY-CLUSTER-01.json

Troubleshooting

Step 3 hangs — nodes not appearing in Foundation Central
Cause: DHCP vendor option (option 43) missing or misconfigured

Log into Foundation Central → Nodes. If nodes are not visible after ~5 minutes of Phoenix boot, the issue is the DHCP vendor option. Verify DHCP is serving IPs on the AHV VLAN. Check that option 43 is present with the correct FC IP and API key. Switch ports must be Access/Native on the AHV VLAN (not tagged) so Phoenix OS gets an untagged DHCP response.

Reference: Foundation Central — DHCP Vendor-Specific Options

Step 1 fails — iLO unreachable (TCP 443 timeout)
Cause: iLO port not connected, wrong VLAN, or IP not assigned

Run Test-NetConnection <iLO_IP> -Port 443 from the jump server. Verify the iLO IP in the config matches the actual iLO IP. Confirm the iLO switch port is on the correct iLO VLAN. Try logging into the iLO web UI (https://<iLO_IP>) from a browser on the jump server.

Pre-flight fails — "IP is responding — already in use"
Cause: Cluster VIP or node IPs still respond from a previous (failed) deployment

The pre-flight check pings all planned IPs to ensure they are free. If they respond, it means a previous deployment left them in place. If you are confident the IPs are stale (e.g. the cluster was already destroyed), use -SkipPreCheck to bypass the gate. Otherwise wait for ARP to expire (~15 min) and re-run.

Step 7 fails — PC registration timeout or unreachable
Cause: Inter-site routing not in place between HUB and remote site

From the jump server, run Test-NetConnection <ClusterVIP> -Port 9440. If that passes but Step 7 still fails, the issue may be that Prism Central itself cannot reach the new cluster over the inter-site WAN link. Verify routing between the HUB site (where PC runs) and the remote site management VLAN.

Step 18 fails — SSH connection refused or wrong password
Cause: TCP 22 blocked, or CVM password changed by Step 15

Run Test-NetConnection <CVM_IP> -Port 22. Check the Step 18 startup banner — it shows which password source was resolved (CSV file from Step 15, config field, or factory default). If Step 15 ran and changed the password, the new password CSV file is used automatically when present in the pipeline scripts directory.

Step 17 fails — dnscmd not found
Cause: DNS RSAT tools not installed on the jump server

Install DNS RSAT on a Windows Server: Add-WindowsFeature -Name RSAT-DNS-Server. On a workstation: Get-WindowsCapability -Name Rsat.Dns.Tools* -Online | Add-WindowsCapability -Online. After installation, verify with Get-Command dnscmd.
