ZTD Deployment Manager

A secure, web-based portal for deploying and managing Nutanix clusters. The app manages authentication, configuration files, and live deployment streaming — PowerShell 7 does the actual work via the automation pipeline.

Node.js · Express HTTPS port 3443 AD / LDAP + local auth WebSocket live streaming Windows Service

What is ZTD Manager

The ZTD Deployment Manager is the web-based front end for the Nutanix Zero Touch Deployment pipeline. It provides a secure browser interface where engineers can build cluster configuration files, trigger deployments, and watch live output — without needing direct access to the jump server or PowerShell.

The app runs as a Windows Service on the jump server. When a deployment is started, it spawns pwsh.exe with Start-Pipeline.ps1 and streams every line of output back to the browser in real time via WebSocket. On completion, the deployment record is saved and a result email is sent.

One tool, one config. The same JSON file that the web UI builds is passed directly to Start-Pipeline.ps1. Every step in the pipeline reads from it — no translation or manual parameter mapping required.

Features

🔐
Secure by default — HTTPS onlyAll traffic is encrypted. The server refuses HTTP connections. Role-based access control with admin and user roles. Bcrypt-hashed local passwords (cost factor 12). Session tokens are server-side and regenerated on each login.
📡
Real-time deployment streamingEvery output line from the PowerShell pipeline streams to the browser live via WebSocket. Step status updates are colour-coded as they happen. The log is retained in full after deployment completes.
⚡
18-step automated pipelineFrom Phoenix boot through NCC health validation. The web UI exposes advanced options: Start At Step (resume from any step), Skip Steps (skip individual steps by number), Dry Run (pre-flight only), and Skip Pre-Check.
💾
Configuration file managementSave and load named JSON cluster configs. The form builds the complete config — no manual JSON editing required. Configs are stored in Nutanix-ZTI/Configs/ and are visible to all authenticated users.
🔑
Active Directory / LDAP SSOConfigure LDAP or LDAPS in the Admin panel. AD users are auto-provisioned on first login and receive a welcome email. All AD users get the user role; promote to admin via Admin → User Management.
📊
Dashboard and audit logFull deployment history with per-step timing averages, success rates, and searchable audit logs. Every login, config change, deployment start/stop, and admin action is logged with timestamp, username, and IP.
📧
Email notificationsPipeline result emails with step timings, NCC health check summary table, and LCM report are sent on completion. Welcome emails are sent when AD users are imported. Both channels use separate SMTP configs.
🪟
Windows Service with auto-restartRuns as a Windows service via WinSW wrapper. Starts automatically on boot and restarts on crash. Managed via services.msc, PowerShell, or the provided npm scripts.

First-Time Setup

💡 Automated setup available. Run .\start.ps1 from PowerShell as Administrator from the deploy-cluster-app\ directory. It installs Node.js, PowerShell 7, npm packages, the SSL certificate, Posh-SSH, and registers the Windows service automatically. The manual steps below are for reference or if you need to set up individual components.
  1. 1

    Install Node.js (v18 LTS or later)

    Open PowerShell as Administrator.

    winget install OpenJS.NodeJS.LTS

    Close and reopen PowerShell after installation. Verify: node --version

  2. 2

    Install PowerShell 7

    The pipeline scripts require pwsh.exe (PowerShell 7.x). Close and reopen PowerShell after installation.

    winget install Microsoft.PowerShell

    Verify: pwsh --version

  3. 3

    Install application dependencies

    cd <path-to>\deploy-cluster-app
npm install
  4. 4

    Create and configure .env

    Copy the example file and edit it. At minimum, set COMPANY_NAME, SESSION_SECRET, and SERVER_URL.

    Copy-Item .env.example .env

    Generate a secure session secret:

    $b = New-Object byte[] 32
[System.Security.Cryptography.RNGCryptoServiceProvider]::new().GetBytes($b)
[Convert]::ToBase64String($b)

    Paste the output as the value for SESSION_SECRET in .env.

  5. 5

    Generate the SSL certificate

    Creates certs\server.key and certs\server.crt using built-in Windows tools — no OpenSSL required.

    .\generate-cert.ps1
  6. 6

    Install Posh-SSH (required for pipeline)

    Steps 15 and 18 of the pipeline use Posh-SSH for SSH connections to CVMs. The pipeline auto-installs it on first run, but installing it explicitly ensures it's available immediately.

    pwsh -Command "Install-Module Posh-SSH -Scope CurrentUser -Force"
  7. 7

    Register as a Windows Service

    This registers the portal as a Windows service so it starts on boot and restarts on crash.

    node install-service.js
Start-Service 'Nutanix Cluster Deployment Web'
Get-Service  'Nutanix Cluster Deployment Web'

    Status should show Running. The service name is Nutanix Cluster Deployment Web.

  8. 8

    First login and credential change

    Browse to https://localhost:3443. Accept the browser warning for the self-signed certificate. Log in with the default admin credentials, then change the password immediately via Admin → User Management.

    Username: admin
Password: Changeme

Company branding

Set COMPANY_NAME in .env — this appears in the nav bar and page title. No server restart is needed after changing COMPANY_NAME (the API reads it dynamically).

SMTP / Email configuration

There are two separate SMTP configurations — they can both point to the same relay:

  • Pipeline result emails (Send-PipelineEmail.ps1): SMTP settings are in .env (SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASS). The recipient is set per-cluster in the JSON config under notify.to.
  • Welcome emails (new AD users): SMTP settings configured in Admin → SMTP Settings, stored in smtp-config.json. This is managed through the web UI without editing files.

Starting a Deployment

On the main page (/), the configuration form builds the JSON file that the pipeline reads. You can either fill it in from scratch or load a previously saved config.

Workflow

  1. Fill in the configuration form — cluster name, node IPs, image URLs, VLAN settings, credentials
  2. Optionally click Save Configuration to save it as a named file for reuse
  3. Expand Advanced Options if you need to set Start At Step, Skip Steps, or Skip Pre-Check
  4. Click Start Deployment to run the full 18-step pipeline, or Dry Run to run pre-flight checks only
  5. Watch live output in the log panel — each pipeline step is prefixed with a colour-coded status indicator
Only one deployment can run at a time. A new deployment request is rejected with HTTP 400 while one is active. Stop the active deployment from the UI before starting a new one.

Advanced options explained

Start At Step

  • Skip steps 1 through N-1 — use when steps are already completed
  • Pre-flight check is also skipped when Start At Step ≥ 5
  • Example: resume from step 7 after a failed PC registration

Skip Steps

  • Comma-separated list of step numbers to skip
  • Useful when a specific step is known to already be complete
  • Example: 9 skips "Create Storage Container" if it already exists

Skip Pre-Check

  • Bypasses the IP availability ping test
  • Use when IPs respond but are confirmed-stale from a previous run

Dry Run

  • Runs only pre-flight checks — no pipeline steps execute
  • Tests iLO access, PC credentials, URL accessibility, IP availability
  • Safe to run anytime, even against a live cluster

Dashboard & Audit Log

Available at /dashboard.html for all authenticated users.

Deployment history

The dashboard shows all deployments from the last 30 days with start time, duration, exit code, cluster name, and triggered-by user. Click any row to view the full deployment log. Step-by-step timing averages are shown for completed multi-step runs.

Statistics

Summary cards at the top of the dashboard show: total deployments (last 30 days), successful count, failed count, and average deployment duration. These update automatically as deployments complete.

Audit log

The audit log records every security-relevant action:

  • Login attempts (successful and failed) with username and source IP
  • Config saves and deletes
  • Deployment starts, stops, and completions
  • Admin actions (user creation, role changes, AD config changes)

Records older than 30 days are automatically purged daily. The log is filterable by date range, username, and action type in the UI.

Admin Panel

Available at /admin.html for admin role only.

User Management

  • Create, edit, and delete local accounts
  • Set roles: admin or user
  • Reset passwords for local accounts
  • Promote AD users imported from LDAP

AD / LDAP Settings

  • Configure LDAP or LDAPS server address and port
  • Set bind user, base DN, and user filter
  • Test AD connectivity before saving
  • Disable AD to revert to local-only auth

SMTP Settings

  • Set SMTP relay for welcome emails
  • Configure sender name and address
  • Send a test email to verify the configuration

Branding

  • Update COMPANY_NAME without editing .env
  • View the server URL and current configuration

Windows Service Management

The service name is Nutanix Cluster Deployment Web. It is managed via PowerShell, services.msc, or the WinSW wrapper.

Day-to-day commands

# Start / Stop / Restart
Start-Service   'Nutanix Cluster Deployment Web'
Stop-Service    'Nutanix Cluster Deployment Web'
Restart-Service 'Nutanix Cluster Deployment Web'

# Check status
Get-Service 'Nutanix Cluster Deployment Web'

Viewing service logs

# Server stdout (deployment output, startup messages)
Get-Content '.\daemon\nutanixclusterdeploymentweb.out.log' -Tail 100 -Wait

# Server stderr (errors and exceptions)
Get-Content '.\daemon\nutanixclusterdeploymentweb.err.log' -Tail 50

# WinSW wrapper (crash/restart events)
Get-Content '.\daemon\nutanixclusterdeploymentweb.wrapper.log' -Tail 30

Uninstall

cd <path-to>\deploy-cluster-app
node uninstall-service.js

Full teardown (lab environments only)

delete.ps1 is the reverse of start.ps1. It removes the service, node_modules, Posh-SSH, Node.js, PowerShell 7, .env, and the SSL certs. It prompts before uninstalling system-level components and does not delete users.json or any cluster configs in Configs/.

.\delete.ps1
⚠ Use delete.ps1 in test/lab environments only. It uninstalls Node.js and PowerShell 7 from the system — these may be used by other tools on the same server.

Security

ControlImplementation
TransportHTTPS only — all HTTP connections are refused. Uses a self-signed certificate by default; replace with a CA-signed cert for production use. Certificate path is configured in .env.
SessionsServer-side sessions with a cryptographically random SESSION_SECRET. Sessions are regenerated on login and invalidated on logout. Session cookie is httpOnly and secure.
PasswordsLocal account passwords are hashed with bcrypt (cost factor 12). Plaintext passwords are never stored in files, logs, or memory beyond the authentication check.
Role enforcementAdmin-only endpoints check the user role server-side on every request — there is no client-side bypass. The user role can start deployments and view the dashboard. Only admin can manage users, delete configs, and change system settings.
Audit loggingAll authentication events, config changes, deployments, and admin actions are written to audit-logs.json with timestamp, username, source IP, and outcome. Records older than 30 days are purged daily.
AD / LDAPLDAP bind uses a service account whose credentials are stored in idp-config.json (only readable by the service process). LDAPS is supported for encrypted directory queries. AD credentials are never logged.
Input handlingConfig JSON submitted via the API is parsed and passed directly to Start-Pipeline.ps1 as a file path — the config is written to disk before execution. No shell interpolation of user input occurs in the spawn call.
